# Safer WebDriver interaction plan

## Goal

Keep the current import/export behavior unchanged while making WebDriver element operations safer against timing issues around visibility, enabled state, and DOM refresh.

## Findings

- `src/utils/webdriver-utils.ts` already protects clicks through `clickElementSafely()`, including wait + scroll + fallback click.
- The same level of protection is missing for textarea input and select updates.
- The highest-risk remaining path is `src/services/emote-handlers.ts`, where the dialogue textarea is manipulated with direct `clear()` / `sendKeys()` immediately after UI state changes.
- `selectOptionSafely()` updates options through script, but does not verify that the owning `select` is ready before doing so.

## Implementation plan

### 1. Centralize readiness checks in `src/utils/webdriver-utils.ts`

- Add a small internal helper that waits until a `WebElement` is both displayed and enabled.
- Keep timeout behavior aligned with the existing click helper so current timing characteristics stay close to today’s behavior.
- Reuse this helper from `clickElementSafely()` instead of duplicating the readiness predicate.

### 2. Make text input safe without changing functional behavior

- Extend the safe input path in `src/utils/webdriver-utils.ts` so input operations wait for interactability before `clear()` and `sendKeys()`.
- Preserve current semantics:
    - still clear the field first,
    - still skip typing when the text is empty or `"（なし）"`,
    - still suppress low-level WebDriver errors inside the utility and let callers keep their current logging/error flow.
- If the helper needs the driver to wait, update its signature in a minimal way rather than adding extra branching at each call site.

### 3. Route current textarea handling through the safe helper

- Update `src/services/emote-handlers.ts` so `DialogueHandler.setEmoteSpecificData()` no longer calls `clear()` / `sendKeys()` directly.
- Keep the existing element lookup order intact: select radio button, locate textarea, write contents.
- Avoid broad control-flow changes; only replace the unsafe element operations with the hardened utility call.

### 4. Harden select updates while keeping the same UI result

- Update `src/utils/webdriver-utils.ts` so `selectOptionSafely()` verifies that the target option and/or parent `select` are in a usable state before setting selection.
- Preserve the current selection mechanism and event dispatch (`input` / `change`) so downstream page behavior does not change.
- Keep the current boolean contract (`true` on success, `false` on failure) and existing caller logging in `src/services/emote-handlers.ts`.

### 5. Reduce stale-element risk where interaction follows a UI mode switch

- Prefer waiting on the element right before the operation, not only when it was first found.
- Keep this inside utility methods where possible so call sites remain simple and behavior stays consistent across click, input, and select flows.

## Files expected to change

- `src/utils/webdriver-utils.ts`
- `src/services/emote-handlers.ts`

## Validation plan

- Run `npm run compile` to catch type regressions from utility signature changes.
- Run `npm run build` to ensure the Electron + renderer bundle still passes.
- Manually smoke test the flows most exposed to the original issue:
    - dialogue edit after opening the modal,
    - stamp edit after switching radio/button state,
    - important-item / other selection after radio toggle,
    - repeated imports to confirm no new retry loops or skipped writes.

## Guardrails

- Do not change business rules, retry counts, logging messages, or success/failure handling semantics unless compilation forces a minimal signature update.
- Do not broaden waits beyond the current helper timeout defaults unless a code-level mismatch requires it.
- Keep changes localized to WebDriver interaction utilities and the direct textarea caller.
