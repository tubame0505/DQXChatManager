## Goal

Refine the emote refactor to keep the new modular structure while fixing the remaining behavioral and robustness issues:

- keep manual-edit tolerance for emote content comparison
- fix double-unescaping behavior in content decoding
- remove the no-op normalization step
- prevent unhandled async failures in IPC handlers
- stop coercing unknown emote types to `"セリフ"`
- harden XPath option lookup against quote-containing text

## Constraints and decisions

- Leave `processSingleEmoteSetting()` retry/result semantics unchanged for now; the user explicitly chose to defer that change.
- Preserve the current refactored architecture (`DqxHiroba` / `EmoteManager` / handlers / utils) rather than folding logic back into `main.ts`.
- Treat leading/trailing spaces as insignificant during content normalization.
- Use a single-pass entity decoding approach so `&amp;lt;` becomes `&lt;` and does not further collapse to `<`.

## Implementation plan

### 1. Unify content normalization and input decoding

Update `src/security/content-sanitizer.ts` to provide shared, explicit content transformation helpers instead of partial ad hoc replacements.

- Replace the current comparison-only chain in `normalizeForComparison()` with ordered normalization rules:
    - convert real `<br>`, `<br/>`, `<br />` tags to newlines first
    - convert `&nbsp;` to plain spaces
    - decode `&lt;` and `&gt;`
    - decode `&amp;` last
    - normalize newline representation
    - `trim()` the result
- Remove the current no-op replacement `replace(/\r\n/g, "\r\n")`.
- Add or restructure helpers so the same semantic rules can be reused by both:
    - comparison logic
    - `sendKeys` preparation
- Keep the distinction between real `<br>` tags and escaped `&lt;br&gt;` text.

### 2. Fix `contentsToKey()` to use the same rules

Update `src/emote-data.ts` so `contentsToKey()` no longer performs its own unsafe replacement chain.

- Route it through the shared sanitizer/normalizer helper.
- Ensure the decode order is safe (`&amp;` last).
- Keep the output appropriate for Selenium input by ending with normalized CRLF-compatible line breaks.

Expected result:

- `&amp;lt;` -> `&lt;`
- `&lt;tag&gt;` -> `<tag>`
- `<br>` -> newline
- `&lt;br&gt;` -> literal `<br>` text

### 3. Keep comparison tolerant but internally consistent

Update comparison call sites in `src/services/emote-manager.ts` only as needed to rely on the unified normalization behavior.

- Continue using normalized comparison rather than strict raw-string comparison.
- Ensure the pre-update skip check and post-update verification both use the same canonical representation.
- Avoid duplicate transformation logic across `EmoteData` and `ContentSanitizer`.

### 4. Fix IPC async error handling in `main.ts`

Update `src/main.ts` to avoid fire-and-forget promise calls for emote import/export.

- Wrap `dqxHiroba.exportEmote()` and `dqxHiroba.importEmote(emote)` with `void ...catch(...)` or equivalent explicit async handling.
- Log failures through the existing `logger` path.
- Keep the current event-based IPC surface unless implementation reveals a strong reason to migrate to `ipcMain.handle`.

Expected result:

- no unhandled promise rejections from import/export
- renderer-triggered failures are still visible in logs

### 5. Remove unsafe unknown-type coercion

Refine emote type parsing in `src/emote-data.ts` and the read path in `src/services/emote-manager.ts`.

- Replace `EmoteData.toEmoteType()` fallback-to-`"セリフ"` behavior with a non-coercing approach.
- Recommended shape:
    - add an `isEmoteType()` guard or `parseEmoteType()` helper that returns `EmoteType | null`
    - keep raw text from the page when the value is unknown
    - only narrow to `EmoteType` once validated
- Update the read path so unknown UI values are not silently rewritten to `"セリフ"`.

Expected result:

- supported values still work unchanged
- unexpected site changes surface as unsupported/unknown instead of being misclassified

### 6. Harden XPath option lookup

Improve `src/utils/webdriver-utils.ts` so `selectOptionSafely()` works even when option text contains quotes.

- Add a small helper to escape arbitrary text into a valid XPath string literal.
- Use that helper when constructing the `option[text()=...]` selector.
- Prefer centralizing the escaping logic in `WebDriverUtils` so all future XPath-by-text lookups can reuse it.

Expected result:

- option lookup remains correct for text containing `'`, `"`, or mixed quoting

## Validation plan

After implementation, run the repository’s recommended checks:

- `npm run compile`
- `npm run build`

Then manually smoke test the emote flow in `npm run dev`:

- import/export still works end-to-end
- dialogue content containing `<br>`, `&nbsp;`, `&lt;`, `&gt;`, `&amp;`, and `&amp;lt;` behaves as expected
- unchanged entries are skipped correctly
- post-update verification still succeeds
- import/export failures log cleanly without unhandled rejection noise
- option selection still works for normal values and any value containing quotes, if such a value exists in the UI

## Files likely to change

- `src/security/content-sanitizer.ts`
- `src/emote-data.ts`
- `src/services/emote-manager.ts`
- `src/main.ts`
- `src/utils/webdriver-utils.ts`

## Notes for implementation

- Prefer small, named helpers over long inline replacement chains.
- Keep normalization semantics easy to reason about: one decoding policy, one comparison policy, shared where possible.
- Avoid broad behavior changes outside the items listed above.
